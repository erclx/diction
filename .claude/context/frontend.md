# Frontend

## Layer responsibilities

- `frontend/src/features/<feature>/` owns one domain slice with its components, hooks, and API client
- `frontend/src/components/ui/` owns vendored shadcn primitives, treated as generated source. `frontend/src/components/` above it holds shared cross-feature UI that is not a primitive, such as `metric-card.tsx` (the score card with its directional-caveat popover) and `level-meter.tsx` (the live capture meter)
- `frontend/src/lib/utils.ts` owns the `cn` class merge helper
- `frontend/src/config.ts` owns the validated `BACKEND_URL`, the only place `import.meta.env` is read
- `frontend/src/test/` owns test infrastructure, MSW server and the provider render helper, not tests
- `frontend/e2e/` owns Playwright specs and the screenshot capture harness. The harness writes captures to `screenshots/<section>/<case>--<theme>.png`, grouped by surface, and includes mobile-width captures for the responsive sidebar drawer. It parks the mouse in the far corner and screenshots with `animations: 'disabled'` so hover fades and the drawer slide freeze at their settled state

## Decisions

- Theme is a manual light, dark, or system choice. `useTheme` in `features/theme/` persists it to the `diction-theme` localStorage key and toggles a `.light` or `.dark` class on `<html>`. The `system` choice clears the class and falls back to `prefers-color-scheme`. An inline script in `index.html` stamps the class before first paint to avoid a flash, and shares the same storage key.
- Tokens live as CSS variables in `:root`, overridden by a `.dark` block for the explicit choice and by a `:root:not(.light):not(.dark)` block inside `prefers-color-scheme: dark` for the system fallback. The two dark blocks hold the same values and must move together. They map to utilities through `@theme inline`, so `bg-background` resolves the active palette at runtime. The shadcn `--sidebar-*` tokens reference the base palette vars (`var(--card)`, `var(--accent)`) rather than holding their own values, so they follow both dark blocks without a third copy. The shadcn install's default `@custom-variant dark` was dropped, since the app themes through these CSS-var swaps rather than `dark:` utilities. One trap follows from that: vendored `components/ui/input.tsx` still carries `dark:` utilities that no longer resolve after the drop. `Input` is unrendered today, reached only through the unused `SidebarInput`, so any surface that renders it must convert those utilities to token classes to keep its dark styling. The first rendered form field, `components/ui/textarea.tsx`, already follows this: it drops the shadcn `dark:` utilities and styles through the base tokens, so it themes through the CSS-var swaps like everything else.
- Type is self-hosted from the Fontsource variable woff2 files. `src/fonts.css` declares the faces with `font-display: optional`, generated from the Fontsource CSS so the file urls stay package-relative and Vite still bundles them. `optional` is the fix for a refresh-time reflow: `swap` painted the larger-metric fallback first, then swapped, so text grew then snapped back. `font-serif` (Newsreader) sets headings and the read-aloud passage. `font-sans` (Source Sans 3) sets UI body and labels. Fonts bundle as local woff2, so the offline guarantee holds.
- A flagged word carries two play controls: its own recorded span through `useSpanPlayer`, and a native reference clip through `useReferenceAudio` in `features/reference-audio/`. The passage card carries the same reference control for the whole passage. The reference hook fetches wav from `GET /api/reference` with TanStack Query keyed on the text and the chosen voice, and plays it through a plain `HTMLAudioElement`, distinct from the Web Audio decode path `useSpanPlayer` needs for unseekable WebM. `useSpanPlayer` widens the played slice with `padSpan`, a fixed 100ms margin each side clamped to the clip, so a slightly-off forced-alignment boundary still contains the flagged word rather than playing silence. `useReferenceAudio` defers the fetch until the first click, so the flagged-word list does not synthesize every word on render.
- The reference voice is an app-wide choice in `features/voice/`. `useVoice` holds it in a module-level store read through `useSyncExternalStore` and persisted to the `diction-voice` localStorage key, so the shell `VoicePicker` and every `useReferenceAudio` and prosody caller share one value without prop drilling. `useVoices` fetches the offered voices from `GET /api/voices`. A null stored voice sends no `voice` param and the backend falls back to its default. `VoicePicker` is a shadcn `Select`, chosen over a native `<select>` so the dropdown carries the app tokens in both themes, and on change it calls `AudioChannel.stop` so a voice switch halts any reference clip already playing. The picked voice also rides the two prosody score requests, so the shadowing and stress scores compare against the voice the user heard.
- Three players sound audio: the reference `HTMLAudioElement`, the `useSpanPlayer` `AudioContext` source, and the own-recording clip through `OwnRecordingAudio`. That wrapper renders a custom themed transport (play and pause `Button`, a primary-filled seek slider, and a mono time readout) over a hidden `HTMLAudioElement` rather than the browser's native `<audio controls>`, which shipped as an un-themed white pill that clashed with the dark cards and read low-contrast on the light background. A single `AudioChannelProvider` in `features/audio-channel/` coordinates them so only one sounds at a time. Each player calls `claim` with its own stop function on play and `release` on end, and `claim` stops whatever else is active. The channel also exposes `stop`, which halts the active player without a new claim, used when the voice changes mid-clip. The coordinator is a claim-and-stop registry, not a playback state machine, so each player still owns its own `isPlaying` flag for its control's active styling. `useSpanPlayer` also exposes the `activeSpan` so a flagged word only shows the playing state for the span that is actually sounding. `ReferenceButton` is a play/stop toggle: a click while its clip plays calls the hook's `stop` rather than replaying, and `useReferenceAudio` releases the channel on the audio `pause` event so a manual stop frees it the same way `ended` does. `useRecorder` also consumes the channel and calls `channel.stop()` at the top of `start`, so beginning a recording on any of the five recording surfaces halts whatever is sounding and the mic never captures playback.
- The capture level meter is driven from `useRecorder`, which taps an `AnalyserNode` off the `MediaRecorder` stream and reads RMS amplitude per animation frame through the pure `computeRmsLevel` in `audio-level.ts`. The `AnalyserNode` and its `AudioContext` tear down on stop and on unmount so no mic stream stays live between reps. Every recording surface renders the shared `LevelMeter` while recording, since they all share the hook.
- The score fetch carries a 60s `AbortSignal.timeout`. GPU scoring is intentionally slow, so the ceiling is generous rather than snappy.
- The app shell in `app.tsx` routes between surfaces with `react-router-dom`, so a refresh keeps the current surface and a session detail is deep-linkable. `BrowserRouter` mounts in `main.tsx` inside the `QueryClientProvider`. The routes are `/` for the passage surface, `/routine` for the suggested-routine home, `/shadowing` for the shadowing surface, `/history` for the session list, `/history/:sessionId` for one session's detail, `/progress` for the progress dashboard, `/drills` for the targeted-drill home, and `/drills/ear-training` and `/drills/production` for the two drills, so `session-history.tsx` reads the selected id from the route param rather than local state. Both drills read an optional `?phoneme=<token>` query param and filter their contrasts to those that train it, which is how the targeted-drill home routes a chosen weak sound into a drill. An absent param keeps a drill's full contrast pool. The minimal-pairs schema, the `useMinimalPairsQuery` hook, and the `filterByPhoneme` and `contrastForPhoneme` helpers live once in `features/minimal-pairs/`, imported by both drills and the phoneme joins, so the shared `['minimal-pairs']` query key has a single set of fetch options rather than a copy per drill. The `?phoneme=` search helper `phonemeSearch` lives in `lib/`, shared by the drill home and the routine so the query convention has one writer. The section title picks the longest matching nav path so `/drills/ear-training` reads as `Ear training` rather than the parent `Targeted`, and the `/drills` nav item matches exactly through an `end` flag. The nav uses `NavLink` inside a shadcn `SidebarMenuButton`, which drives `aria-current="page"` from the active route, and the active item takes a `sidebar-accent` fill through an `aria-[current=page]` variant rather than the primitive's own `data-active` prop. `react-router-dom` was chosen over TanStack Router to avoid a second routing-and-loader paradigm, and the router owns URL state only while TanStack Query keeps all fetching.
- The shell is the shadcn `Sidebar` primitive at its 16rem default, `collapsible="icon"` so it folds to an icon rail on desktop and becomes a `Sheet` drawer below the 768px mobile breakpoint from `useIsMobile`. Nav items sit in three `SidebarGroup`s, a `Practice` group led by the `Routine` suggestion surface with `Passage` at `/` still the app landing, a `Drills` group, and a `Review` group, each item carrying a lucide icon. A slim top bar in `SidebarInset` holds the `SidebarTrigger`, the current section title, and the backend status, the reference `VoicePicker`, and the theme toggle on the right. The top bar is `sticky top-0` with a translucent backdrop so it stays in place as a surface scrolls. The status drops its `Backend: <state>` label below the `sm` breakpoint. Collapse state persists to a `sidebar_state` cookie, and `Ctrl` or `Cmd` plus `B` toggles it. `SidebarProvider` reads `window.matchMedia`, stubbed in `src/test/setup.ts` so jsdom renders the shell.
- The suggested routine in `features/routine/` sequences the built modes into one ordered sitting rather than leaving mode choice manual. A `modes.ts` registry is the single source for each mode's label, icon, route, and whether it takes a target phoneme, read by both the pure `buildRoutine` planner and the surface, so a new mode is one row. Free-topic is absent until v0.7 joins it as one row. `buildRoutine` leads with the due sounds as alternating production and ear-training drill steps, interleaves the fixed-content modes so no two adjacent steps share a mode, caps the sitting at `ROUTINE_LIMIT`, and falls back to the weak-sound ranking then a fixed passage-shadowing-stress starter rotation, mirroring the drill home's due-first-else-weak fallback. It drops any due or weak phoneme with no curated minimal pair before weaving, so a sound the drills cannot train falls through to the fixed rotation rather than linking a dead-end drill, the same case the drill home guards. `useRoutine` composes the same three queries as `useSuggestedDrills` and degrades a resurfacing error to the weak-sound path rather than OR-ing it into `isError`, the resilience the drill home follows. The routine is a stateless suggestion recomputed on each visit with no completion state, matching the recompute-on-read pattern, so tracked progression stays a follow-up.
- The passage and free-topic surfaces let the user practice their own text. While the recorder is idle each renders its content as an editable `Textarea` seeded with a default, and locks it to static text once recording starts, so the scored text cannot change mid-capture. Both share one pure validator, `validatePracticeText` in `lib/practice-text.ts`, which trims the input and returns a single error string when the text is empty, has no letters, or exceeds a per-surface length cap. The trimmed value is the one source fed to scoring, and on the passage surface also to the reference button and its `disabled` gate, so display, playback, and scoring never diverge. An error disables the record control with the message shown inline under the field, and `ReferenceButton` halts a clip it is already playing when its `disabled` prop goes true, so reference audio never outlives the text it belongs to. The free-topic `Shuffle` control cycles the fixed `FREE_TOPIC_PROMPTS` as suggestions, the offline default the plan keeps alongside the typed path. The passage surface adds a third source beside the default and the typed path: `useGeneratePassage` POSTs to `/api/content/generate` and seeds the returned text into the same `passage` state through the mutation's `onSuccess`, so generated text flows through `validatePracticeText` and scores like typed text. A `Focus on my weak sounds` checkbox drives the bias: when on, the top `FOCUS_PHONEME_COUNT` phonemes from `useWeakSoundsQuery` ride the request as `focus_phonemes`, and when off an empty list asks for a general passage. The checkbox stays disabled with a hint until the tracker holds at least one sound, so flipping it on can never silently yield an unbiased passage. It carries the project focus ring rather than the native checkbox outline. The control renders only while idle, so a generation never lands under a stale score, and a failed request shows an inline error while leaving the current text untouched. Generated shadowing and stress lines stay a later phase, since those surfaces couple generation to reference-audio synthesis.
- Heading sizes step down from the shell: surface titles are `text-2xl`, sub-headings `text-xl`, and score numbers `text-2xl` to match across the list and detail. The read-aloud passage is `text-lg`, sized down so it does not compete with the surface title.

## Hidden contracts

- The score response schema in `score-result.ts` mirrors the backend contract in `.claude/context/scoring.md`. The two must move together.
- `ClipTooWeakError` marks the 422 boundary, defined once in `use-score-passage.ts` and reused by the word, shadowing, and prosody-analyze score paths. The UI branches on it to offer a re-record instead of a retry, and the prosody surfaces render its `detail` as the too-weak reason, so any new score-path error must preserve that type.
- Mic-driven e2e specs are gated to chromium because fake media relies on chromium launch flags set on that project in `playwright.config.ts`.

## Gotchas

- `MediaRecorder` assembles the blob in its async `onstop` handler, so the recording appears one tick after `stop()`. The object URL is revoked in a `useEffect` cleanup keyed on the recording.
- `MediaRecorder` WebM clips are not seekable. `duration` reads `Infinity` and setting `HTMLAudioElement.currentTime` fails silently, so seeking to a flagged span played no audio. `useSpanPlayer` decodes the blob into an `AudioBuffer` and plays exact ranges through `AudioBufferSourceNode.start(0, start, end - start)`, which sidesteps seeking entirely. The `OwnRecordingAudio` transport plays the whole clip through a plain `HTMLAudioElement`, where whole-clip seeking does work once duration resolves, so it forces `Infinity` duration to a real value by seeking to `Number.MAX_SAFE_INTEGER` on `loadedmetadata` and reading `duration` back on the next `timeupdate`.
- lint-staged does not prettier-format `.ts` files, only `.md` and `.json`. CI `check:format` runs prettier over everything, so a `.ts` file can pass commit and fail CI. Run prettier before pushing.
- cspell still matches zero files from a linked worktree because `.claude/worktrees/` is gitignored. `check:spell` passes `--no-must-find-files` so `bun run check` and pre-push no longer fail on the empty match, but spelling goes genuinely unchecked there. Verify by pointing cspell at an out-of-tree config that sets `useGitignore: false` with the project dictionaries.
- e2e text assertions on short words collide with the passage copy. `getByText('thought')` matched `thoughtful`, so exact matching is required for flagged-word checks.
